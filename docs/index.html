Article I

Vision & philosophy

LifeOS is not a collection of apps. It is a personal operating system — every module shares one database, one automation engine, one user model, one event log, and one task system. A new feature integrates into that shared ecosystem; it does not stand alone.

Why LifeOS exists

Productivity today is fragmented across tools that don't talk to each other: Google Calendar holds events, Google Tasks holds tasks, Notion holds notes, GitHub holds projects, PDFs sit in Drive, expenses live in a spreadsheet somewhere. Each tool is fine on its own — the cost is in the seams between them. LifeOS exists to be the orchestration layer that connects them: one place that understands a task, a document, an expense, and a calendar event as parts of the same life, not seven unrelated records.

Guiding principles

These are the yardsticks every future design decision gets measured against. They're expanded into full immutable rules — with rationale and good/bad examples — in Article XXII, Engineering Principles.

01

Supabase is the single source of truth. External services mirror data; they do not own it.

02

Modules are interconnected. No module evolves into a silo with its own isolated data model if a shared model already exists.

03

Automation before manual work. If a repetitive task can be automated reliably, LifeOS should automate it.

04

Deterministic logic before AI. Use rules where rules are sufficient; use AI only when interpretation or generation is genuinely needed.

05

Build complete vertical slices. Finish one usable capability end-to-end — database → workflow → integration → documentation — before starting another.

06

Optimize for maintainability. Favor clear, modular workflows over clever but hard-to-understand implementations.

07

Every architectural decision is documented. If a future developer asks "why was it built this way?", the answer already exists in this guide — see Article IV, Architecture Decision Records.

Article IV

Architecture decision records

The historical record of why LifeOS is built the way it is. New architectural decisions get appended here, in order, and are never rewritten after the fact — if a decision is later reversed, that reversal is its own new ADR that supersedes the old one.

ADR 001

Telegram instead of a native mobile app

accepted
Reason

Rapid development, cross-platform by default, native push notifications, and zero app-store install friction — a chat interface was reachable on day one.

Trade-offs

UI is constrained to text/buttons; no custom native screens; every module is bottlenecked through one webhook (see ADR 004).

Alternatives considered
  • Native iOS/Android app
  • Web app (PWA)
  • WhatsApp Business API
Future impact

The Phase 10 React dashboard is additive, not a replacement — Telegram stays the primary interface.

ADR 002

Supabase is the single source of truth

accepted
Reason

One Postgres schema that every module reads and writes is what makes AnalyticsOS able to query across modules later. Splitting storage per module would make that impossible without a sync layer of its own.

Trade-offs

Supabase becomes a single point of failure (see Article XVIII, Failure Strategy) and a scaling bottleneck if usage ever grows past a single user.

Alternatives considered
  • Per-module databases
  • Firebase / Firestore
  • Self-hosted Postgres without Supabase's auth/storage layer
Future impact

Every future integration (Calendar, dashboard, exports) is defined as a mirror of Supabase, never a second writer — see ADR 003.

ADR 003

Google Calendar is synchronization only

accepted
Reason

Calendar's data model (events with start/end times) can't represent a task, a grade, or an expense. Making it the database would force every other module to be reshaped around Calendar's constraints instead of its own.

Trade-offs

Requires a sync workflow and accepts the possibility of drift if a user edits an event directly in Google Calendar.

Alternatives considered
  • Google Calendar as primary store for events
  • Two-way real-time sync
Future impact

Conflict resolution rule: Supabase always wins. A Calendar-side edit is either pulled back in explicitly or overwritten on the next sync — this still needs to be formalized before Phase 03 ships.

ADR 004

One Telegram Gateway, never per-module triggers

accepted
Reason

A Telegram bot can register exactly one webhook URL — a platform limit, not a preference. Every module has to funnel through a single router (Article VIII).

Trade-offs

The Gateway workflow becomes a critical path — if it goes down, every module goes down with it, not just one.

Alternatives considered
  • Separate bots per module (rejected — fragments the user experience across N bots)
  • Polling instead of webhooks (rejected — higher latency, no real benefit)
Future impact

This is why the Router (Phase 02) has to ship before a second command exists — retrofitting it later means rebuilding every workflow that came before it.

ADR 005

n8n for orchestration

accepted
Reason

Workflow automation gives visual, inspectable pipelines for wiring Telegram → logic → database → external services, without hand-writing the same request/response plumbing seven times.

Trade-offs

Large Code nodes become unmaintainable inside n8n's editor (see Article XXII, Coding Standards, Rule 6) — n8n is meant to orchestrate, not host the application.

Alternatives considered
  • Hand-rolled Node.js backend with no visual workflow layer
  • Zapier / Make (rejected — not self-hostable, ongoing cost)
Future impact

If a Code node ever exceeds ~50 lines, that logic graduates into a proper backend service that n8n calls, rather than growing in place.

ADR 006

AI never stores state

accepted
Reason

Gemini is used for interpretation and generation — classifying intent, summarizing a PDF, generating a quiz — never for holding or being the record of what's true. That job belongs to Supabase and deterministic SQL. See Article XIII, AI Philosophy, for the full boundary.

Trade-offs

Slightly more workflow steps per AI-assisted action (classify → structure → validate → write), instead of letting the model "just handle it."

Alternatives considered
  • Letting Gemini call Supabase functions directly
  • Using AI-managed memory/state instead of a schema
Future impact

Every AI response the system acts on has to be structured JSON, never free text — enforced at the Router (Article VIII, Rule 1).

Article VIII — load-bearing

The Router

A Telegram bot can register exactly one webhook URL. That's a platform limit, not a style choice — it means every module funnels through a single router workflow, never through its own independent Telegram Trigger node. (Not to be confused with the changelog's v0.2.0 "Telegram Gateway" — see the note in Article V.)

Telegram Trigger — lives only in 001-Gateway
↓ starts with "/" ?
Yes — literal command

Direct switch on the command string. No AI call, no latency, no ambiguity.

/addtask /tasks /done /delete /quiz /apply /budget
No — natural language

Gemini classifies intent and returns structured JSON, never free text:

{"intent":"add_task","title":"Finish biology","due_date":"2026-07-17","priority":"high"}

Unknown / low-confidence → Gateway replies with help text instead of guessing.

both paths resolve to the same intent key ↓
Execute Workflow → 2XX-Tasks / 3XX-StudyOS / 7XX-FinanceOS…
R1

One intent vocabulary

A literal command and its natural-language equivalent must resolve to the identical intent key (e.g. /addtask and "remind me to…" both → add_task). The router doesn't care which path it came from.

R2

Sub-workflows own their reply

Per the philosophy pipeline, the module workflow — not the Gateway — sends the final Telegram reply. The Gateway only routes.

R3

No silent failure

An unrecognized command or a low-confidence Gemini classification must produce a reply. "Nothing happens" is the one failure mode that isn't allowed.

Telegram command specification

One page listing every command that reaches the Router. This is a registry, not a duplicate spec — validation, workflow, DB writes, and events are each owned by the module the command routes to, per its Module Contract (Article XIV) and the Event Registry (Article XII). This section exists so the full command surface is visible in one place.

Article XII

Event log architecture

A single events_log table that every module writes to whenever something meaningful happens. This is what lets AnalyticsOS (Article XIV) read one table instead of querying all seven modules directly.

Naming collision to watch: CoreOS already has a table called events (Article X) for calendar events — start_time, end_time, source. That is not this table. events_log is an append-only activity trail across all modules; events is calendar state owned by CoreOS. Keep the _log suffix to keep them apart, same rule as the Gateway/Router split in Article VII (Versioning).
ColumnPurpose
idUUID primary key
user_idFK → users.id
modulewhich module emitted it — core, study, finance, career, projects, analytics
event_typee.g. task_created, quiz_completed
payloadjsonb — event-specific detail, kept small
created_attimestamp

This table is additive, not a replacement for the domain tables — tasks is still the source of truth for a task's current state. events_log is the append-only history of what happened, which a mutable row can't tell you on its own.

Event registry

Every event gets an owner before it gets an implementation. This is the contract — producer, consumers, and retry behavior — not the payload shape. Payloads are marked TBD until the owning module actually ships and defines them; writing a shape now would be inventing detail the module hasn't decided yet.

Article XIII

AI philosophy

Gemini has exactly three jobs in LifeOS. Everything deterministic stays deterministic — see Guiding Principle 04 in Article I, and ADR 006 in Article IV.

①

Interpret

Turn unstructured input into structured intent — "remind me to submit the assignment friday" → {"intent":"add_task","due_date":"2026-07-17"}.

②

Generate

Produce new content from existing data — PDF summaries, quiz questions, weekly report prose.

③

Recommend

Surface a suggestion for the user to accept or ignore — task prioritization, schedule changes, resume gap analysis. AI never auto-applies a recommendation without the user (or an explicit rule) confirming it.

✓ Good uses
Extract due date · Classify intent
Summarize PDF · Generate quiz
Prioritize tasks (suggestion only) · Resume gap analysis
✕ AI never does this
Store data — that's Supabase's job
Authenticate users
Replace SQL for reads or writes
Manage workflow state or control flow
Be the source of truth for anything

Article XVII

Development philosophy

How work actually gets sequenced, day to day — the operating habits behind the roadmap in Article XXI.

01

Build vertically. Don't build five unfinished modules — finish one feature completely (Article XVI) before starting the next.

02

Every sprint produces usable software. Not a partial workflow, not a schema with no UI in front of it — something that can actually be used through Telegram.

03

Prefer maintainability over cleverness. A boring workflow a stranger can follow beats a clever one only the author understands.

04

Architecture before implementation. Design (step 1 of Article XVI) happens before a single node gets placed.

05

Infrastructure should never dictate architecture. The Quick Tunnel being dev-only doesn't change how a module is designed — it just delays when it can run unattended (see Article XX, Known Gaps).

06

Shared components over duplicated logic. If two modules need the same reply-formatting or validation logic, it goes in workflows/shared/, not copy-pasted twice.

Article XIX

Failure strategy

Balances transparency with the user against not spamming them about every transient hiccup — while protecting data integrity above both.

R1

Never fail silently

Every failure resolves to one of: user notification, retry, event log entry, or escalation. "Nothing happens" is forbidden — this extends Article VIII Rule 3 (Router) to every workflow, not just the Router.

R2

Never claim success on a failed write

If a database write fails, the reply must say so. "Task created" when the insert failed is a worse bug than any crash — this is the core data-integrity rule.

User-facing workflows
Telegram command · task creation · Gemini call
On failure: log it → reply to the user → stop cleanly
"I couldn't create your task because Google Calendar is unavailable. Nothing has been lost — try again shortly."
Never expose stack traces or raw API errors
Background jobs
Daily summaries · calendar sync · reminders · DB cleanup
On failure: log the event → retry automatically if appropriate
Notify the user only if the failure becomes persistent or touches their data
No Telegram message for every background hiccup

Retry policy

Failure typeExamplesBehavior
TransientHTTP 429, timeouts, temporary network issues, 503Retry with exponential backoff
PermanentInvalid API key, missing permissions, malformed request, auth errorFail immediately · log · notify user if action was user-initiated

Service-specific behavior

ServiceOn failure
TelegramRetry the send; if it still fails, log the event for later inspection.
GeminiReturn a friendly fallback — "I'm having trouble thinking right now, please try again shortly" — never a raw API error.
Google CalendarDo not pretend the event was created. Tell the user the calendar was not updated.
SupabaseAbort the operation. Never continue with a partially written row.
n8nLog the execution, preserve context where possible, avoid leaving the system in an inconsistent state.

Standard rollback strategy — applies to every phase

1. Restore the previous workflow export from workflows/. 2. Restore the previous database migration (down-migration or backup restore). 3. Redeploy the previous Docker image / compose config. 4. Retag the previous version if the broken tag already reached main (Article VII, Rule 3 — tags are never renamed, a new patch tag is cut instead).

JavaScript (Code nodes & backend services)

RuleConvention
Language levelES2023+ · const by default · let only when reassignment is required · never var

SQL & database design

RuleConvention
Tables & columnssnake_case — see Article IX for the full naming table
Primary keyAlways id UUID. Never a bare integer ID.
Foreign keysAlways explicit — user_id, task_id, project_id
Standard columnsEvery table carries id, created_at, updated_at. User-owned tables always carry user_id.
EnumsPrefer CHECK constraints over native Postgres ENUM types until there's a compelling reason otherwise.

n8n workflow & node naming

✕ Never
If · If1 · Code · Code2
✓ Always
Check Existing User · Route Command · Parse Intent

You should know exactly what a node does without opening it. Workflows follow ModuleOS - NNN - Description, e.g. CoreOS - 003 - Task Engine — numbers never change once assigned.

①

One responsibility per workflow

Don't build Telegram → Everything. Build Gateway → Router → Task Engine → Calendar Engine as separate workflows, each doing one job.

②

Code nodes stay tiny

~50 lines max. Past that, the logic moves into a proper backend service that n8n calls — n8n orchestrates, it doesn't become the application (ADR 005).

③

Handle every external failure

Telegram, Supabase, Gemini, Google Calendar, Cloudflare — never assume success. See Article XVIII, Failure Strategy, for the full policy.

④

Log for analytics later

Every workflow should eventually log success, failure, duration, user, and operation — this is what Article XII's event log is built to capture.
